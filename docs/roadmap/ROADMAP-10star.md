# repo-librarian: the 10-star roadmap

Generated by CEO review, 2026-07-08. Mode: SCOPE EXPANSION. Automation model: Approach B (trust-tiered), phased.
Constraint that overrides everything: **added autonomy and generation must not erode accuracy.** That is the brand.

## Accepted scope (10 workstreams)

Committed baseline (from Chris's issues):
- B1 Visibility fix — nudge fires on real work-resumption, not cold-start only
- B2 One-command apply — machine-applyable proposals + `librarian apply`
- B3 Archive command + retirement detection — `librarian archive` + a new dream job type
- B4 Trust-ladder autonomy — `[automation]` config, conservative by default
- B5 Enrichment engine — gap detection -> source query -> provisional doc draft
- B6 Vocab cleanup — kb -> librarian/catalog/stacks

Dream-big adds (all four accepted):
- E1 MCP query-on-demand server — sourced, freshness-stamped answers without preloading CATALOG.md
- E2 Auto-generated verify checks — enrichment writes the check that keeps its own fact honest
- E3 Provenance chain — `librarian why <fact>`: command + source + timestamp for every verified fact
- E4 Librarian PR bot — dream branch auto-opens a PR; approval = review-UI checkboxes

## The two foundational primitives (build once, reused by almost everything)

1. **Proposal object** — a versioned JSON schema: `{type: fix|archive|read_when|merge|enrich, target, action/patch, rationale, provenance}`.
   The dream agent emits these instead of prose. `librarian apply` consumes them. The trust-ladder decides which auto-apply.
   The PR bot renders them as checkboxes. **B2, B3, B4, E4 all depend on this. It is the spine.**
   Treat the schema as a compatibility surface (like STALENESS.md line 3). Version it.

2. **Provenance record** — `{fact, command, source, extracted_value, timestamp, doc, line}`.
   verify.py already computes most of this at CheckResult; it just isn't persisted as a queryable chain.
   **E3 (why), E2 (auto-verify), E1 (MCP sourced answers), B5 (enrichment labels) all consume it.**

## Build order (dependency-correct)

**Phase 0 — spine (do first, cheap while touching everything):**
- Proposal object schema + persistence
- Provenance record + persistence (_index/provenance.json)
- B6 vocab cleanup (fold in while editing assets)

**Phase 1 — packaged commands, HITL everywhere (makes the tool retainable):**
- B1 visibility fix (UserPromptSubmit hook) — MOVED FIRST (eng-review: zero dependency on the spine, pure retention win). Hook MUST fast-path: stat a timestamp file and early-exit in <10ms BEFORE loading catalog.json, else every user prompt pays the catalog-load tax (eng-review P1).
- B2 `librarian apply` (consumes proposals, reindexes, marks done — one command)
- B3 `librarian archive <path>` (frontmatter flip + git mv + reindex, atomic, reversible)
- B3 retirement detection (new dream job type: provably-merged patches, shipped handoffs, done plans) — PROPOSE-ONLY always
- **E1a `librarian query --json` (pure-stdlib CLI) — PULLED FORWARD from Phase 3** (eng-review D2): delivers the whole token-cost-flip value with zero deps, and de-risks the core value hypothesis early instead of backloading it to the last phase.

**Phase 2 — autonomy + generation (the leap):**
- B4 trust-ladder ([automation] config, default = propose-only for everything; auto-apply strictly opt-in, branch-only unless commit explicitly enabled)
- B5 enrichment engine (gap detection -> source query -> provisional draft, always labeled + source-carried + quarantined with a decay TTL)
- E2 auto-generated verify checks (couples to B5; checks land in _index/generated-checks.json, NOT hand-emitted TOML — eng-review D3)
- E3 provenance query (`librarian why`)

**Phase 3 — surfaces (how the value reaches agents and teams):**
- E1b MCP server — as an OPTIONAL EXTRA (`pip install repo-librarian[mcp]`, `[project.optional-dependencies]`), thin wrapper over the Phase-1 `librarian query`. Core install stays zero-dep (eng-review D2).
- E4 Librarian PR bot — GitHub via the `gh` CLI or stdlib urllib, NOT a third-party Python GitHub lib (keeps zero-dep).

## Accuracy-preservation guarantees (non-negotiable)

- R1 Enrichment hallucination: output ALWAYS labeled provisional + carries the source query (verified-by-construction). E2 auto-writes the recurring check. Degenerate guard: source returns empty/zero -> do NOT draft "we have zero X"; flag "source empty, gap unconfirmed."
- R2 Wrong-tier auto-apply: trust-ladder conservative by default; auto-apply opt-in per job type, branch-only unless commit explicitly enabled. Strict config validation (extend config.py unknown-key = error to [automation]).
- R3 Retirement false-positive: propose-only ALWAYS (never auto-tier); requires positive evidence (patch content present in HEAD, or explicit supersedes pointer); archive is reversible (git mv + status flip), never delete.
- R4 Nudge fatigue: throttle to once per work-block; only surface when actionable (due worklist OR stale verify); never a heartbeat.
- R5 MCP staleness (the RAG-that-lies trap): answers carry freshness metadata; verified facts can re-verify on demand or stamp age loudly.
- R6 Proposal schema churn: get it right early, version it, it is a compatibility surface.

## Observability (first-class, not afterthought)

- _index/apply-log.jsonl — every apply records what changed
- enrichment logs source + query + result
- doctor gains: automation-tier config validity, orphaned proposals, enrichment sources reachable

## Proving it's worth the tokens (answers Chris's benchmark question)

The scope changes the benchmark strategy. Stop competing on tasks grep can do.
- New ENRICHMENT benchmark class: a question the corpus cannot answer. Bare agent scores 0; librarian queries the source and scores 1. That is 0->1, not 2x-for-parity.
- New MCP on-demand accounting mode: tokens for "answer 5 questions" with always-load catalog vs on-demand MCP. Show the crossover where on-demand wins.
- Keep the honest ethos: publish where it still costs more.
